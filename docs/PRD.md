# PRD: 개인용 러닝 기록 앱 (MVP)
프로젝트 이름: 복슬달리기(BokslRunning)

## 1. 목적

* 사용자가 러닝을 **기록(경로/시간/거리/페이스)** 하고,
* **누적 통계와 월별 차트**를 확인하며,
* 언제든 **전체 데이터를 파일로 내보내기** 할 수 있는 개인용 앱을 만든다.

## 2. 타깃 사용자

* 본인 1명(개인용)
* 러닝 기록을 장기 보관하고, 다른 앱으로 이동/백업 가능한 형태를 원함

## 3. 핵심 사용자 시나리오

1. 사용자는 처음 실행 시 **몸무게/성별/나이**를 입력한다.
2. 사용자는 **러닝 시작 → 러닝 중 확인 → 종료** 흐름으로 기록한다.
3. 사용자는 기록 목록/상세에서 과거 기록을 확인한다.
4. 사용자는 누적(거리/시간/평균 속도)과 월별 차트를 확인한다.
5. 사용자는 “전체 내보내기”로 모든 기록을 파일로 저장/공유한다.

---

# 4. 기능 요구사항 (고객 관점)

## A. 프로필

* A1. 사용자는 **몸무게(kg), 성별, 나이**를 입력/수정할 수 있어야 한다.
* A2. 프로필이 입력되지 않으면 칼로리는 “입력 시 계산 가능”으로 안내하거나 표시하지 않는다.

## B. 러닝 기록 및 트래킹

* B1. 사용자는 러닝을 **시작/종료**할 수 있어야 한다.
* B2. 러닝 중 앱은 다음 값을 **실시간으로 표시**해야 한다.

  * 러닝 시간
  * 거리
  * 현재 페이스(또는 속도)
  * 평균 페이스(또는 평균 속도)
  * 최고 속도
  * 칼로리(추정)
* B3. 앱은 러닝 **GPS 경로(지도 동선)**를 기록해야 한다.
* B4. 러닝 종료 시 앱은 **요약 화면**을 제공해야 한다.

  * 거리/시간/평균 페이스/최고 속도/칼로리 + 지도 경로

## C. 오프라인 기록

* C1. 사용자는 **네트워크가 없어도** 러닝을 시작/기록/종료할 수 있어야 한다.
* C2. 오프라인에서 생성된 기록도 앱 내에서 **저장되어 재열람** 가능해야 한다.

## D. 기록 열람

* D1. 사용자는 기록 목록에서 러닝 기록을 날짜순으로 볼 수 있어야 한다.
* D2. 사용자는 기록 상세에서 **지도 경로**와 **요약 수치**를 볼 수 있어야 한다.

## E. 분석·통계

* E1. 사용자는 **전체 누적** 기준으로 **거리/시간/평균 속도**를 확인할 수 있어야 한다.
* E2. 사용자는 **월별 차트**(거리/시간/평균 속도)를 확인할 수 있어야 한다.
* E3. 사용자는 통계 차트를 **좌우 스와이프**하여 더 과거 월 데이터를 확인할 수 있어야 한다.

## F. 전체 데이터 내보내기

* F1. 사용자는 지금까지의 모든 러닝 기록을 **한 번에 내보내기** 할 수 있어야 한다.
* F2. 내보내기 결과는 사용자가 보관/이동할 수 있도록 **파일 형태**여야 한다.
* F3. 내보내기 데이터에는 최소한 아래 정보가 포함되어야 한다.

  * 모든 러닝 세션의 요약(거리/시간/평균/최고/칼로리/시각)
  * 각 러닝 세션의 GPS 경로(시간 순 트랙 포인트)
* F4. MVP 내보내기 파일 형식은 `JSON 1개`로 정의한다. (예: `bokslrunning_export_v1.json`)
* F5. 내보내기 파일에는 `schema_version` 필드를 포함한다.

## G. 데이터 가져오기

* G1. 사용자는 내보내기 파일(JSON)을 앱으로 **가져오기** 할 수 있어야 한다.
* G2. 가져오기 시작 전, 앱은 현재 데이터를 **내부 백업**해야 한다.
* G3. 가져오기 방식은 **병합**을 기본 정책으로 한다. (기존 데이터 유지 + 가져온 데이터 추가)
* G4. 가져오기 완료 후, 사용자에게 병합 결과(예: 추가/중복 건수)를 안내해야 한다.
* G5. 동일한 내보내기 파일을 다시 가져오면 **중복 가져오기를 수행하지 않고 안내**해야 한다.

---

# 5. 품질 요구사항 (고객 체감 기준)

* Q1. 앱은 러닝 기록 중 갑작스런 앱 종료/중단이 발생해도, **마지막 저장 지점까지의 기록**은 잃지 않아야 한다.
* Q2. 앱은 위치 권한이 없거나 GPS가 비활성일 때 사용자가 이해할 수 있도록 안내해야 한다.
* Q3. 앱은 러닝 중 화면이 꺼지는 일반적인 사용 상황에서도 기록이 지속되어야 한다. *(가능 범위 내에서 “기록 지속”을 목표로 함)*
* Q4. 앱은 러닝 중 **OS에 의한 백그라운드 프로세스 종료가 발생하지 않도록 가능한 범위에서 최소화**해야 한다.

# 6. 비범위 (MVP에서 제외)

* 계정/로그인, 클라우드 동기화
* 소셜(팔로우/피드/공유 커뮤니티)
* 고도/상승고도 분석
* 워치/심박 등 외부 센서 연동
* 코스 생성/내비/세그먼트

---

# 7. 성공 기준 (자가 테스트 기준)

* S1. 10분 러닝을 기록하면, 목록에 저장되고 상세에서 지도 경로가 보인다.
* S2. 비행기 모드(오프라인)로 10분 기록해도 저장되고 재열람된다.
* S3. 통계 화면의 누적/월별 차트 값이 기록들과 일치한다.
* S4. 전체 내보내기를 실행하면 파일 1개가 생성되고, 기록 전체가 포함된다.
* S5. 가져오기를 실행하면 기존 데이터가 내부 백업된 뒤 병합되고, 결과 안내가 표시된다.

---

# 8. 화면 구성(최소)

* 홈(러닝 시작, 누적 요약)
* 러닝 화면(실시간 수치 + 지도 + 종료)
* 종료 요약 화면
* 기록 목록 / 기록 상세
* 통계(누적 요약 + 월별 차트)
* 설정(프로필, 전체 내보내기, 가져오기)

---

# 9. 정책/용어 정의 (MVP)

* P1. `저장`은 러닝 세션을 기록으로 확정하는 행동이다.
* P2. 종료 요약 화면의 `완료`는 저장 확정이 아니라 화면 닫기/이동 행동이다.
* P3. `러닝 시간`은 시작부터 종료까지의 시간이다.
* P4. 앱이 중단된 뒤 재실행하면, 이전 세션은 **마지막 저장 지점부터 이어서 기록**한다. 중단된 동안의 구간은 공백으로 남을 수 있다.
