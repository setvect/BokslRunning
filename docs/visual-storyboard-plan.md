# Visual Storyboard Plan (BokslRunning MVP)

## 1. 목적

개발 전에 핵심 사용자 흐름을 시각적으로 검증한다.
- 화면 배치/정보 우선순위 확인
- 버튼 흐름/실수 가능성 확인
- 상태별(권한/GPS/오프라인/중단 후 재실행) 대응 확인

---

## 2. 작업 방식 (3단계)

### Step 1. 저해상도 와이어프레임 (빠르게)
- 도구: Excalidraw, FigJam 중 택1
- 산출물: 화면별 러프 스케치
- 원칙: 컬러/타이포 디테일 금지, 정보 구조만 확인

### Step 2. 중간충실도 UI
- 도구: Figma
- 산출물: 모바일 기준 화면 세트
- 원칙: 실제 버튼 라벨/상태 텍스트 반영

### Step 3. 클릭 프로토타입
- 도구: Figma Prototype
- 산출물: 주요 플로우 클릭 데모
- 원칙: 핵심 3개 플로우부터 연결

---

## 3. 우선 제작 플로우

### Flow A. 첫 실행
온보딩 -> 프로필 입력 -> 위치 권한 -> 홈

### Flow B. 러닝 기록
홈 -> Ready(GPS) -> 러닝 중 -> 종료 확인 -> 요약 -> 목록/상세

### Flow C. 중단 후 이어하기
앱 중단/재실행 -> 이어서 기록 안내 -> 이어하기 또는 폐기

---

## 6. 러닝 앱 전용 UI 가이드 (간단 버전)

- 러닝 중 화면은 한 손 조작 우선: `종료` 버튼은 하단 고정
- 지도보다 지표 우선순위를 명확히: 시간/거리/페이스를 상단 배치
- 상태 메시지는 짧고 즉시 이해 가능하게: 예) "GPS 신호 약함"
- 중단 후 재실행 화면에서는 "공백 구간 가능"을 반드시 고지

---

## 7. 리뷰 체크리스트

### 사용성
- [ ] 처음 사용자도 10초 안에 `러닝 시작` 위치를 찾을 수 있는가?
- [ ] 러닝 중 현재 상태(기록 중/종료 직전)를 혼동하지 않는가?
- [ ] 오탭으로 기록이 사라질 위험이 없는가?

### 일관성
- [ ] `저장`, `완료`, `이어하기`, `폐기` 의미가 화면마다 동일한가?
- [ ] PRD 용어와 UI 문구가 동일한가?

### 예외 처리
- [ ] 권한 거부 시 사용자 행동 경로가 있는가?
- [ ] 오프라인/GPS 불량에서도 핵심 기록 흐름이 유지되는가?
- [ ] 앱 재실행 시 마지막 저장 지점부터 이어지는 흐름이 보이는가?

---

## 8. 완료 기준 (디자인 단계)

- [ ] 3개 핵심 플로우가 클릭 프로토타입으로 연결됨
- [ ] 화면별 상태(정상/오프라인/GPS 약함/이어하기)가 최소 1개 이상 반영됨
- [ ] 문구/버튼 라벨이 PRD, storyboard와 충돌 없음
- [ ] 리뷰 피드백 1회 이상 반영됨

---

## 9. 바로 시작 순서 (권장)

1. `storyboard.md` 기준으로 Step 1 와이어프레임 15장 작성
2. Flow B(러닝 기록)만 먼저 중간충실도로 고도화
3. 종료/저장/완료/이어하기 라벨 확정
4. Flow A, C까지 프로토타입 연결
